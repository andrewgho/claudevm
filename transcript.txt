❯ This project is to set up scripts and cookbook documentation to enable the    
following scenario. With a single command (something like `claudebox create     
$name`), I can create (on this MacBook Pro running Apple Silicon) a brand new   
VM with identifier $name. This VM is a Debian Linux VM which does not mount any 
 of the VM host's directories, but does get outbound Internet access and a      
virtual local IP so that if it runs a service, I can hit it from the MacBook    
Pro which is the VM host. This MacBook Pro already has Colima, QEMU (probably   
unnecessary since ARM Linux is fine for now), and Docker (also probably         
unnecessary unless that seems like the cleanest way to accomplish the goals     
below).                                                                         
                                                                                
The VM should be sized for general development (this MacBook Pro has 32GB of    
RAM, so we could generously make it 4GB or even 8GB), and runs an up to date    
Debian headless server distribution that is minimal, but has common build and   
development tools (compiler, common languages, Git, Emacs and vim). It has a    
single non-root user named claude who has passwordless access to sudo (for      
example, the user account is in group wheel, and /etc/sudoers has `Defaults     
!lecture, !requiretty, shell_noargs, timestamp_timeout=300` and `%wheel         
ALL=(ALL) NOPASSWD: ALL`). The user has shell /bin/bash and has a ~/bin         
directory in its $PATH. Its home directory is prepopulated, for example, by     
copying in a tarball from the VM host on VM creation; this could be how the     
$PATH setting works. Claude Code for terminal should be preinstalled.           
                                                                                
The user should be able to run `claudebox create $name` and then the VM is      
created, and its claude home directory populated. The user should then be       
dropped into an interactive session which is in subdirectory ~/work and execs   
Claude Code. If the user runs `claudebox connect $name` (or some other command, 
 which could be a direct VM layer command documented in a cookbook) in another  
terminal window, they get another interactive session that goes to the bash     
prompt.                                                                         
                                                                                
Ideally, if I do some command (maybe `claudebox claude $name`?) that starts a   
Claude Code session in an interactive window, as with `claudebox create $name`. 
 Even more ideally, if a Claude Code isn't already running on the VM, then      
`claudebox claude $name` should reconnect to the previously disconnected        
session there (this implies we need to record the session ID somewhere? or      
maybe there is some other recipe to reconnect reliably).                        
                                                                                
There should be similar recipes or script frontends to suspend and tear down    
the VM.                                                                         
                                                                                
The goal is to run Claude Code in a mode where it doesn't ever need to ask me   
for permissions (all permissions set, as with                                   
`--allow-dangerously-skip-permissions` and `--dangerously-skip-permissions`),   
but, because Claude is running a VM and not on my actual host, it is okay if    
Claude destroys the VM environment or otherwise causes a problem. I can just    
tear down the VM and recreate it.

⏺ FIRST REQUEST: Existing Systems

  No single existing tool does all of this out of the box. Here's what's
  closest:

  Lima (limactl) — the strongest building block, and you almost certainly have
  it already (Colima depends on it). Named instances, Debian support,
  configurable mounts (disable completely with --mount-none), port forwarding,
  persistent SSH. Gaps: no native "drop into Claude Code" session management, no
   home tarball population, no session reconnection story.

  Orbstack — most polished CLI ergonomics (orb create, orb shell), named
  instances, direct VM IP from host (no port forwarding needed). Gaps:
  Ubuntu-focused (no official Debian), can't fully disable mounts, commercial
  product.

  Multipass — clean multipass launch --name foo ergonomics, bridged networking.
  Gap: Ubuntu only, no Debian.

  Tart — Apple Virtualization.framework, very fast ARM VMs, CI-focused. Gap:
  designed for ephemeral CI runs, not interactive dev sessions, IP discovery
  unreliable.

  Claude Cowork (Anthropic's official feature in Claude Desktop) — most
  relevant, uses Apple Virtualization.framework + bubblewrap sandboxing per
  session. Gaps: Ubuntu only, network allowlisted to package registries +
  Anthropic API, session dies if the Desktop app closes (no reconnection),
  single shared VM not named instances.

  Shuru — ephemeral AI agent sandbox, one-command setup, integrates as a Claude
  skill. Gap: intentionally ephemeral (rootfs resets), no persistent dev
  sessions.

  Bottom line: Lima gives you ~60% of the way there. The remaining 40% — the
  claudebox wrapper, home directory tarball, Claude Code preinstall, and
  tmux-based session reconnection — doesn't exist as a package anywhere.

  SECOND REQUEST: Build Plan

  The stack: Lima as the VM layer (already installed via Colima), cloud-init for
   provisioning, tmux for session reconnection, and a claudebox bash script
  tying it together.

  Architecture

  ~/.claudebox/
    bin/claudebox          # main CLI script (symlinked to ~/bin or
  /usr/local/bin)
    template.yaml          # Lima VM template (Debian, no mounts, cloud-init)
    home.tar.gz            # seed tarball for ~/claude home dir (~/.bashrc,
  ~/bin/, etc.)

  Lima stores VM state in ~/.lima/<name>/. Each named VM is fully independent.

  ---
  VM Template (template.yaml)

  A Lima YAML file specifying:
  - Image: Debian 12 (Bookworm) arm64 cloud image (official
  debian-12-generic-arm64.qcow2)
  - Resources: 4–8 GB RAM, 4 CPUs, 40 GB disk
  - Mounts: none (mounts: [])
  - Network: Lima with socket_vmnet gives the VM a real IP reachable from the
  host
  - Provision: inline cloud-init script (runs once on first boot)

  Cloud-init Provisioning (runs once at VM creation)

  1. Create user claude, shell /bin/bash, add to group wheel
  2. Write /etc/sudoers.d/wheel: Defaults 
  !lecture,!requiretty,timestamp_timeout=300 + %wheel ALL=(ALL) NOPASSWD: ALL
  3. apt-get install -y: build-essential git emacs-nox vim curl wget tmux 
  python3 python3-pip golang nodejs npm ca-certificates
  4. Install Claude Code: npm install -g @anthropic-ai/claude-code (as claude
  user)
  5. Create ~/work directory
  6. Extract the home tarball (shipped separately, copied over SSH after VM
  first boots)

  The home tarball provides: .bashrc (with ~/bin in $PATH), .profile,
  .bash_aliases, ~/bin/ skeleton, and any dotfiles the user wants seeded.

  ---
  claudebox Script Subcommands

  claudebox create <name>    # create+provision VM, populate home, drop into
  Claude Code
  claudebox claude <name>    # attach to (or start) Claude Code tmux session
  claudebox connect <name>   # interactive bash shell in the VM
  claudebox suspend <name>   # limactl stop <name>
  claudebox resume <name>    # limactl start <name>
  claudebox destroy <name>   # limactl delete <name> (with confirmation prompt)
  claudebox list             # limactl list
  claudebox ip <name>        # print VM's IP address

  ---
  Session Reconnection (the key insight)

  tmux's new-session -A -s <name> flag: attach if the session exists, create if
  it doesn't. So:

  # claudebox claude <name>:
  ssh -t claude@$(claudebox ip $name) \
    "tmux new-session -A -s claude -c ~/work -- \
     claude --dangerously-skip-permissions"

  - First call: creates tmux session claude, starts Claude Code in ~/work
  - After SSH disconnect: tmux session keeps running in the VM, Claude Code
  continues
  - Next call: reattaches to the same session — you see exactly where Claude
  left off
  - If Claude Code exited: creates a fresh Claude Code session in the same tmux
  session

  This means claudebox create and claudebox claude do the same SSH command;
  create just runs provisioning first.

  For claudebox connect, skip the tmux+claude wrapper and just SSH to a plain
  bash shell:

  ssh claude@$(claudebox ip $name)

  ---
  Networking

  Lima with socket_vmnet (which Colima already uses) assigns the VM an IP on a
  host-only network (e.g., 192.168.105.x). Services running in the VM are
  reachable directly from your Mac at that IP. No port forwarding needed.
  claudebox ip <name> wraps limactl shell <name> -- hostname -I | awk '{print 
  $1}' or parses Lima's SSH config.

  3. home.tar.gz — your dotfiles seed (.bashrc with PATH, ~/bin/, etc.)
  4. claudebox bash script — ~200 lines wrapping limactl, SSH, tmux

  lifting; the script is mostly glue. Want me to build it?
